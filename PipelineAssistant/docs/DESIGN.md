# UE5 Asset Validation Pipeline

> 技术设计文档（Technical Design）

---

# 1. 项目概述

在 UE 项目开发过程中，随着资源数量不断增加，仅依赖人工检查容易出现模型面数超标、LOD 缺失、贴图规格不统一以及资源引用异常等问题。这类问题通常需要美术、TA 或程序在资源进入项目后再逐个排查，不仅效率较低，也容易遗漏。

本项目基于 UE5 Python 开发了一套轻量级资产校验工具，用于对项目资源进行批量扫描，根据预设规则自动检测资源是否符合规范，并将检测结果导出为 CSV 报告，帮助开发人员快速定位问题资源。

当前版本主要针对 StaticMesh、Texture、Lightmap UV 以及 Hard Reference 四类资源规则进行检测，并支持通过配置文件管理检测标准，以适配不同项目的资源规范。

---

# 2. 设计原则

整个工具开发过程中主要遵循以下三个原则。

## 2.1 配置与规则分离

检测规则负责实现校验逻辑，检测标准统一由 `AuditRules.json` 管理。不同项目只需调整配置文件即可修改检测标准，无需修改 Python 代码。

## 2.2 优先保证稳定性

对于当前 UE Python API 无法稳定支持的能力，例如 UV 几何分析、Import Pipeline 接入等，不采用复杂绕行方案，而是在文档中明确记录限制，保证工具行为可预期。

## 2.3 保持轻量化

工具完全基于 UE5 Python 实现，不依赖 C++ 插件开发。各类检测规则保持独立，便于后续继续扩展新的检测项。

---

# 3. 功能范围

## 当前支持

### StaticMesh

- Triangle 数量检测（LOD0）
- Vertex Count 检测（LOD0）
- LOD 层级数量检测

### Texture

- 最大分辨率检测
- 是否为 2 次幂尺寸检测
- 压缩格式检测（根据资源命名约定判断期望的 Compression Settings，并与资源当前设置进行比较）

### Lightmap

- Lightmap UV 通道数量检测

### Reference

- Hard Reference 递归依赖检测
- Hard Reference 缺失资源检测

### 其它功能

- JSON 配置规则
- CSV 检测报告导出
- Window 菜单工具入口
- 自动初始化
- 热重载按钮
- 白名单豁免机制

---

## 当前暂不支持

由于当前版本采用 UE Python 实现，以下能力暂未纳入本项目：

- Soft Reference 检测
- Import Pipeline 自动校验
- UV 重叠、0~1 范围等几何级检测
- 自动修复功能

---

# 4. 整体流程

工具整体执行流程如下：

```text
启动工具
    │
    ▼
确定扫描范围
（资产 / 文件夹 / 全项目）
    │
    ▼
读取 AuditRules.json
    │
    ▼
遍历资产
    │
    ▼
根据资源类型执行对应规则
    │
    ▼
收集检测结果
    │
    ▼
导出 CSV 报告
```

扫描范围采用统一入口自动判断。

优先级如下：

1. 当前选中资产
2. 当前选中文件夹
3. 全项目（`/Game`）

若同时存在多种选择，则按照以上优先级执行。

工具在每次开始扫描前都会重新读取 `AuditRules.json`，因此修改配置文件后无需重新启动工具即可生效。

# 5. 关键设计

为了实现一套便于维护、易于扩展的资产校验工具，开发过程中主要围绕开发语言、规则组织、配置管理以及检测范围四个方面进行了设计。

## 5.1 为什么选择 Python

本项目定位为编辑器工具，因此优先考虑开发效率。

UE 官方提供了较完整的 Python API，可以访问 Asset Registry、ToolMenus 等编辑器功能，对于资源扫描和规则检测已经能够满足需求。

相比 C++，Python 开发周期更短，也更方便后续继续增加新的检测规则。

---

## 5.2 为什么没有使用 Data Validation Framework

项目初期曾尝试使用 UE 自带的 Data Validation Framework。

调研并实际验证后发现，Python 可以调用已有 Validation 功能，但无法注册自定义 Validation Rule，无法满足本项目需要的扩展能力。

因此最终放弃该方案，改为基于 Asset Registry 遍历资源，再使用 Python 执行各项检测规则。

这种方式虽然需要自行维护规则调度，但能够满足当前工具的实际需求，同时保持较好的扩展性。

---

## 5.3 为什么使用 JSON 配置规则

不同项目对于角色、道具、植被等资源通常采用不同的规范，因此没有将检测阈值直接写入代码，而是统一放入 `AuditRules.json` 中管理。

当前版本支持针对 Character、Prop、Vegetation 三类资源分别配置 Triangle、Vertex 以及 LOD 标准，扫描时由调用方指定当前资源类别。

配置文件结构示例如下：

```json
{
    "StaticMesh": {
        "Character": {
            "MaxTriangles": 50000,
            "MaxVertices": 80000,
            "MinLODCount": 2
        }
    },
    "Texture": {
        "MaxSize": 2048
    },
    "LightmapUV": {
        "RequireChannel2": true
    }
}
```

不同项目可根据资源规范调整配置，无需修改检测规则代码。

白名单同样配置于 `AuditRules.json` 中。

当前版本仅针对指定目录跳过 LOD 检测，采用目录前缀匹配方式，配置目录下的子资源自动生效。

---

## 5.4 为什么采用 Hard Reference 递归检测

当前版本仅检测 Hard Reference。

检测过程中，工具首先获取资源的直接依赖关系，再继续递归分析下一层依赖，直到达到设定的最大递归深度。

当前默认最大递归深度为 **10 层**。
**该深度为代码常量，目前未开放配置。**

为了避免循环引用导致重复检测，工具在递归过程中维护已访问资源集合，同一个资源不会被重复分析。

当前版本未包含 Soft Reference 检测，因为 Soft Reference 更多用于运行时资源加载策略，不属于本工具当前的静态资源规范检查范围。

# 6. 开发过程中遇到的问题

整个开发过程中主要遇到了以下四个问题。

## 问题一：放弃 Data Validation Framework

### 问题

项目初期计划直接使用 UE 自带的 Data Validation Framework，实现与编辑器统一的资源校验流程。

### 分析

查阅官方文档并进行实际验证后发现，Python 可以调用已有 Validation 功能，但无法注册自定义 Validation Rule，无法满足本项目需要的扩展能力。

这属于当前 Python API 的能力限制，而不是实现方式的问题。

### 方案

放弃 Data Validation Framework。

改为基于 Asset Registry 遍历项目资源，再使用 Python 执行各项检测规则。

### 结果

工具完全基于 Python 实现，规则扩展更加灵活，也符合当前项目开发范围。

---

## 问题二：增加 Vertex Count 检测

### 问题

最初仅计划检测 Triangle 数量。

开发过程中了解到，美术制作过程中经常出现 Triangle 数量正常，但 Vertex 数量明显偏高的情况。

### 分析

进一步学习渲染流程后了解到，模型导入引擎后会因为 UV 接缝、硬边等原因发生 Vertex Split，Vertex Count 更能反映模型实际渲染成本，因此增加了 Vertex Count 检测。

### 方案

在 Triangle 检测基础上，增加 Vertex Count 检测，并针对不同资源类型分别配置对应阈值。

### 结果

最终工具同时检测 Triangle 与 Vertex 两项指标，使检测结果更加符合实际项目需求。

---

## 问题三：Lightmap UV 检测能力受 Python API 限制

### 问题

原计划检测 Lightmap UV 是否存在重叠、是否位于 0~1 空间等更完整的规范。

### 分析

实际开发过程中发现，当前 Python API 无法访问 StaticMesh 的底层几何数据，因此无法完成几何级分析。

### 方案

调整检测目标，仅检测是否存在 Lightmap UV 通道。

### 结果

当前版本完成了 Lightmap UV 通道检测，其余几何级检测暂未纳入本版本范围。

---

## 问题四：LOD 检测与实际项目存在冲突

### 问题

LOD 检测规则完成后，部分原型资源、临时资源全部被判定为不符合规范。

### 分析

结合实际项目使用情况，这类资源通常不会进入正式版本，因此无需按照正式资源标准进行检测。

### 方案

增加白名单机制，对指定目录下的资源跳过 LOD 检测。

白名单统一配置于 `AuditRules.json`，便于不同项目根据实际情况调整。

### 结果

降低了无效告警数量，使检测结果更加符合实际生产流程。

---

# 7. 最终实现与使用方式

工具提供统一入口，根据当前编辑器状态自动选择扫描方式。工具即可根据当前编辑器状态自动选择扫描方式。

支持以下三种扫描模式：

| 扫描方式       | 说明                         |
| -------------- | ---------------------------- |
| 当前选中资产   | 检测一个或多个选中的资源     |
| 当前选中文件夹 | 递归检测当前文件夹内所有资源 |
| 全项目扫描     | 默认递归扫描 `/Game`         |

扫描过程中统一读取 `AuditRules.json` 配置，根据资源类型执行对应检测规则，并最终生成 CSV 报告。

另外，为减少原型资源、临时资源带来的误报，工具支持白名单配置，可针对指定目录跳过部分检测规则。

CSV 报告包含以下字段：

| 字段                 | 说明         |
| -------------------- | ------------ |
| Asset Path           | 资源完整路径 |
| Asset Name           | 资源名称     |
| Asset Class          | 资源类型     |
| Rule                 | 触发规则     |
| Severity             | 问题等级     |
| Current Value        | 当前检测值   |
| Threshold / Standard | 对应规范     |
| Message              | 问题说明     |
| Suggestion           | 修改建议     |

Severity 定义如下：

| 等级    | 含义                         |
| ------- | ---------------------------- |
| Error   | 不符合当前资源规范，需要修改 |
| Warning | 建议优化，不影响资源正常使用 |

检测规则采用模块化组织方式，各资源类型对应独立规则文件，方便后续继续增加新的检测项。

# 8. 测试验证

工具完成后，对各项规则进行了实际验证。

### StaticMesh

复制引擎自带 `SM_Cube`，构造仅包含一个 LOD 的测试模型，验证 LOD 检测规则能够正确识别缺失情况。

同时验证 Triangle 与 Vertex Count 是否能够按照配置阈值进行检测。

### Texture

使用 Windows 画图创建 1000 × 1000 PNG 图片并导入项目，验证非 2 次幂尺寸检测。

同时验证命名规则对应的 Compression Settings 是否能够正确匹配。

### Hard Reference

创建测试 Material，并引用一张 Texture。

随后删除该 Texture，验证 Hard Reference 缺失检测是否能够正确识别失效引用。

同时验证递归依赖检测能够正常遍历资源引用关系。

### Lightmap UV

使用仅包含一个 UV 通道的 StaticMesh，验证 Lightmap UV 通道检测规则。

上述测试均能够正确输出检测结果，并生成对应 CSV 报告。


# 9. 当前限制

受当前开发范围及 UE Python API 能力限制，当前版本仍存在以下限制。

## 9.1 Lightmap UV 检测能力有限

当前版本仅检测资源是否包含 Lightmap UV 通道。

对于 UV 是否存在重叠、是否位于 0~1 空间、UV Island 是否符合规范等几何级分析，由于当前 Python API 无法访问相关底层数据，因此暂未实现。

## 9.2 当前仅检测 Hard Reference

工具目前仅检测 Hard Reference。

Soft Reference 更多用于运行时资源加载策略，不属于当前版本的静态资源规范检查范围，因此未纳入本项目。

---

## 9.3 无法接入资源导入流程

当前版本无法接入 Import Pipeline，在资源导入时自动执行检测。

这是由于 UE Python 当前无法完成相关扩展能力，因此工具采用手动扫描方式完成资源检查。

---

## 9.4 当前仅提供检测能力

本工具仅负责发现问题并生成检测报告。

对于资源修复、LOD 自动生成、贴图压缩修改等操作，均需要开发人员或美术人员根据报告自行处理，当前版本未提供自动修复功能。

---

# 10. 后续优化

结合当前开发情况，后续计划继续完善以下几个方向。

## 增加更多资源类型

目前工具主要针对 StaticMesh、Texture、Lightmap UV 以及 Hard Reference 进行检测。

后续计划逐步增加 SkeletalMesh、Animation 等资源类型的检测规则。

---

## 优化检测报告

当前版本采用 CSV 导出检测结果。

后续可增加 HTML 等更直观的报告形式，方便查看统计信息及问题分类。

---

## 优化扫描体验

当前扫描过程中采用同步执行方式。

当资源数量较多时，扫描时间可能较长。

后续可增加扫描进度提示，使工具使用过程更加直观。

---

## 研究 Import Pipeline 扩展方案

若后续采用 C++ 开发，可进一步研究接入 Import Pipeline，在资源导入阶段自动执行资源规范检查，减少问题资源进入项目。
